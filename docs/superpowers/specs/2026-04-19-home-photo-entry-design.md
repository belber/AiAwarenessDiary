# 首页图片入口交互调整设计

## 目标

将首页输入框的“上传图片”入口从文件选择器改为更接近系统相册的体验，同时保留快速拍照能力。点击入口后不再直接进入 `OpenDocument`，而是先展示一个轻量底部菜单，允许用户选择 `从相册选择` 或 `拍照`。

## 当前问题

- 首页当前使用 `ActivityResultContracts.OpenDocument()`，交互更像文件管理器，不像用户预期的系统相册。
- 用户希望“上传图片”优先进入照片相关流程，而不是文档选择流程。
- 用户也希望在相同入口下保留快速拍照能力，而不是额外增加一个常驻拍照按钮继续挤压输入区。

## 方案

### 入口交互

- 点击首页输入框右侧的图片按钮后，不再立即打开文件选择器。
- 改为弹出一个简洁的底部菜单，提供两个动作：
  - `从相册选择`
  - `拍照`
- 不新增第三个常驻按钮，保持现有输入区结构不变。

### 选择相册

- `从相册选择` 改用系统照片选择器 `ActivityResultContracts.PickVisualMedia()`。
- 媒体类型限制为 `ImageOnly`。
- 若用户取消选择，不报错、不清空已有待发送图片。
- 选中图片后，仍复用现有 `viewModel.importPendingPhoto(uri)` 流程，最终进入当前输入区的待发送图片预览。

### 拍照

- `拍照` 使用系统相机 `ActivityResultContracts.TakePicture()`。
- 在拉起相机前创建临时文件与对应 `Uri`，拍摄成功后通过现有 `setPendingCapturedPhoto(...)` 流程写入 `pendingCapturedPhotoFile`。
- 拍照完成后不跳转页面，直接回到首页输入区并展示待发送图片预览。
- 若用户取消拍照，不清空已有待发送图片。

### 状态与替换规则

- 相册选择和拍照最终都汇合到现有的 pending photo 状态，不改变保存记录时的主流程。
- 若当前已经存在一张待发送图片，新图片覆盖旧图片。
- 覆盖时沿用现有临时文件清理逻辑，避免残留无用缓存文件。

## 代码边界

- `InputScreen.kt`
  - 将 `OpenDocument` 替换为底部菜单 + `PickVisualMedia` / `TakePicture` launcher 接线。
  - 管理图片入口菜单显示状态。
  - 在拍照失败、没有可用相机等情况下触发 snackbar。
- `MainViewModel.kt`
  - 复用现有 `importPendingPhoto(...)`、`setPendingCapturedPhoto(...)`、临时文件替换与清理逻辑。
  - 如有必要，补充一个创建拍照临时文件的入口或更清晰的状态更新 helper，但不改动保存记录主流程。
- 照片存储相关类
  - 若当前已具备拍照临时文件创建能力则直接复用；若没有，则在现有 `RecordPhotoStorage` 范围内补齐。

## 失败处理

- 相册取消：静默返回，不提示错误。
- 拍照取消：静默返回，不提示错误。
- 拍照返回失败：提示 snackbar，例如 `拍照失败，请重试`。
- 没有可用相机或临时文件创建失败：通过现有错误通道提示用户。
- 读取或持久化图片失败：沿用当前 `uiState.error` 机制展示 snackbar。

## 测试范围

- `MainViewModel` 测试：
  - 相册导入后正确进入 pending 状态。
  - 拍照图片写入后正确进入 pending 状态。
  - 替换 pending 图片时旧临时文件被清理。
- 首页 helper 测试：
  - 图片入口文案与入口模式相关 helper 的回归验证。
- 本次不新增 UI instrumentation test，先以 ViewModel 和纯逻辑测试保证回归风险可控。

## 验收标准

- 点击首页图片按钮后，优先看到一个底部菜单，而不是文件选择器。
- 选择 `从相册选择` 时，进入系统照片选择器而不是文档选择器。
- 选择 `拍照` 时，可直接调起系统相机并在成功后回到首页展示图片预览。
- 相册与拍照两条路径都能复用现有待发送图片和保存记录流程。
- 取消选择或取消拍照不会误清空当前已选图片。
